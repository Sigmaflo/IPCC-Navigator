# 📝 Technical Thought Log: IPCC RAG Reboot

## 1. 프로젝트 재정립 (Project Pivot)
- **이유:** 챗봇 서비스에서 가장 중요한 것은 답변의 '신뢰성'임. 기존에는 사용자 인터페이스와 경험(UX)에 집중했으나, IPCC 보고서와 같은 전문 도메인에서는 오정보 전달이 치명적임을 인지함.
- **결정:** 데이터 전처리(Preprocessing)와 답변의 신뢰성 확보를 최우선 순위로 설정. "동작하는 챗봇"보다 "믿을 수 있는 지식 베이스" 구축에 집중함.

## 2. 현재까지의 진행상황 (Current Status)
- **성능 측정:** 100개의 데이터셋에 대해 RAGAS로 **Recall(재현율)** 지표를 측정한 결과 평균 **0.8381** 달성 (성공 데이터 80개 기준).
- **인사이트:** 신뢰성의 첫걸음은 Retrieval이 얼마나 정확한 문서를 가져오는가에 달려있다고 판단하여 Recall을 첫 핵심 지표로 선정함. 이후 Context Precision(검색 결과 상위 노출도) 측정 예정.
- **문제 분석 (NaN 데이터):** 전체 데이터 중 20개에서 NaN 발생. 이는 Gemma 3.1:8B 모델의 컨텍스트 처리 한계 및 입력 토큰 길이 초과로 인한 추론 실패로 예상됨.
- **데이터셋 전략:**
    - 0.8381은 낮지 않은 수치이나, 데이터셋 재구성을 통해 청크 길이를 최적화하고 재측정할 예정.
    - 질문의 다양화와 데이터 길이 축소 방안 고민 중.
    - **NotebookLM 활용 유지:** RAGAS 자동 생성보다 NotebookLM이 문서 맥락을 더 잘 이해하여 고품질의 골든 데이터셋을 뽑아낸다고 판단. 번거롭더라도 현재 방식 유지.

## 3. 예정 계획 (Future Roadmap)
- **데이터셋 고도화:** 데이터셋 재구성 및 Recall 지표 재측정.
- **언어 실험:** 현재 SR6 한글 번역본을 사용 중이나, RAG 시스템의 영문 원문 추론 성능이 더 높을 것으로 예상됨. **원문 vs 번역본 성능 비교 실험** 진행.
- **로컬 최적화:** 4600H 노트북 환경에서 실행하기 위한 모델 경량화 시도.
- **환경 구축:** Windows 환경에서 **WSL2** 기반 개발 환경 구축 및 관리.
- **엔진 검증:** **Ollama**를 우선 사용하여 성능을 검증하고, 이후 필요에 따라 llama.cpp와 리소스 점유율 및 추론 속도 비교 분석.
