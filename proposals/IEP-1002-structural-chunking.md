## 동기

IEP-1001에서 단순 청킹 실험 결과 CASE 3(1000/200)과 CASE 4(1200/240)가 각각 0.8520, 0.8567로 최고 성능을 기록하였으나, 두 가지 한계가 확인되었다.

첫째, **사실 확인 유형**의 Context Recall이 CASE 3 기준 0.8627로 개선되었음에도, 고정 길이 방식의 구조적 한계(문맥 단절)가 근본적으로 해소되지 않았다. 둘째, CASE 4(1200)에서 NaN이 15개로 급증하여 청크 크기를 단순히 키우는 방식의 한계가 드러났다.

IPCC 보고서는 `섹션 → 숫자점숫자(X.X) → 숫자점숫자점숫자(X.X.X)` 3단계 계층 구조로 명확하게 조직되어 있다. 이 구조를 청킹 경계로 활용하면 임의적인 문맥 단절을 줄이고, 특히 **비교·의견/예측 유형**에서 성능 향상이 기대된다.

**발견된 문제: 본문 헤딩 보완 실패**

목차 텍스트에는 trailing 대시가 포함되어 있어(`2.1 관측된 변화, 영향 및 속성 ---`) 본문 텍스트와 문자열이 달랐다. 이를 정제한 클린 제목으로 본문을 재스캔했으나 **0개 탐지**. 원인은 pdfplumber가 `2.1 관측된 변화` 같은 절 제목을 독립된 줄이 아닌 앞뒤 문장과 연결해 추출하기 때문이다. 결과적으로 `숫자점숫자` 32개, `숫자점숫자점숫자` 28개가 청킹 경계에서 누락되었다.

#### Step 2: 청킹 로직 구현 및 실행

**Fallback 설계**

| 조건 | 처리 방식 |
| :--- | :--- |
| 청크 크기 < 100자 | 다음 청크와 병합 (`merge_min`) |
| 청크 크기 > 2,000자 | 문장 경계(마침표+개행)에서 분할 (`split_max`) |

**청킹 결과**

| 항목 | 수치 | 비고 |
| :--- | :---: | :--- |
| 원시 청크 수 | 48개 | 헤딩 경계 기준 |
| 원시 청크 평균 크기 | ~10,000자 | 헤딩 48개로 구간이 너무 큼 |
| 최종 청크 수 | 284개 | fallback 적용 후 |
| 평균 청크 크기 | 1,669자 | |
| 중앙값 | 1,918자 | IEP-1001 최적 구간(1,000~1,200자)보다 큼 |
| 최소 / 최대 | 36자 / 2,000자 | |
| fallback 비율 | **99.3%** | split_max 279개, merge_min 3개 |

> **fallback 99.3%의 의미**: 헤딩 48개로 원시 청크 평균이 ~10,000자에 달해 전부 `split_max`로 분할됨. 실질적으로 "헤딩 메타데이터가 붙은 고정 길이 청킹"에 가까우며, 구조 기반 청킹 본래의 의도(문맥 단절 방지)는 달성되지 않았다.

## 분석

_Day 3 완료 후 최종 작성 예정_

**Day 2까지의 중간 분석:**

구조 기반 청킹의 핵심 전제는 "헤딩 경계 사이 텍스트를 하나의 의미 단위로 추출한다"는 것이다. 그러나 pdfplumber의 텍스트 추출 방식으로 인해 절 단위 헤딩(`숫자점숫자`, `숫자점숫자점숫자`)이 본문에서 독립적으로 탐지되지 않았고, 실질적으로 활용 가능한 헤딩이 48개에 그쳤다. 이는 원래 설계의 60개 헤딩(목차 기준)에도 못 미치는 수준이다.

결과적으로 원시 청크 평균이 ~10,000자로 커져 99.3%가 `split_max`로 분할되었으며, 이는 "구조에 따른 청킹"이 아닌 "구조 메타데이터만 붙은 고정 길이 청킹"이 되었다. Day 3 평가에서 IEP-1001 대비 유의미한 성능 차이가 나타나기 어려운 구조다.
