## 동기

IEP-1002에서 구조 기반 청킹을 시도했으나 전체 Context Recall이 0.7106으로 IEP-1001 CASE 3(0.8520) 대비 **-0.1414** 하락했다. 실패의 근본 원인은 pdfplumber가 절 단위 헤딩(`X.X`, `X.X.X`)을 독립 줄로 추출하지 못해 헤딩 48개만 확보되었고, 원시 청크 평균이 ~10,000자에 달해 fallback(split_max)이 99.3%에 달한 것이다. 결과적으로 "헤딩 메타데이터만 붙은 고정 길이 청킹"으로 전락했다.

SemanticChunker(LangChain)는 헤딩 탐지에 의존하지 않고 **문장 간 임베딩 유사도 변화**로 청크 경계를 결정한다. 이는 IEP-1002의 pdfplumber 제약을 직접 우회하며, 의미 단위로 텍스트를 분리하므로 사실 확인 유형(IEP-1002 Recall 0.3333)의 구조적 약점을 해소할 가능성이 높다.

> **주의**: SemanticChunker도 pdfplumber 추출 텍스트를 입력으로 받는다.
> 헤딩 탐지 의존성은 우회하지만, 2단 레이아웃 합침·중복 텍스트 등 원문 추출 품질 문제는
> Phase 3 IEP-1004(파서 교체)에서 해소한다.

## 분석

### threshold 선택 — 평균보다 중앙값이 신뢰할 수 있는 대표값

SemanticChunker p95의 평균(2,481자)과 중앙값(1,280자)의 괴리가 크다. 이는 소수의 초대형 청크(최대 27,236자)가 평균을 왜곡하기 때문이다. 중앙값 1,280자는 IEP-1001 최적 구간(1,000~1,200자)에 가장 근접하며, 100자 미만 노이즈 청크도 17개로 3종 중 최소다. 평균이 아닌 중앙값을 기준으로 threshold를 선택한 것이 이번 실험의 핵심 판단이다.

### NaN 85개 — 후처리에도 불구하고 감소하지 않음

5,000자 상한 후처리로 NaN 감소를 기대했으나 실제 NaN은 85개로 IEP-1002(34개)보다 오히려 증가했다. 원인은 두 가지다. 첫째, 후처리 후에도 평균 청크 크기가 2,096자로 IEP-1001 CASE 3(1,000자)의 두 배 이상이다. top-3 청크를 컨텍스트로 넘기면 최대 6,000자+가 되어 `llama3.1:8b` 처리 한계를 초과한다. 둘째, pdfplumber 텍스트의 노이즈(빈 줄, 도표 잔재)가 SemanticChunker 경계 결정을 교란하여 내용상 불완전한 청크가 다수 생성되었고, 이것이 RAGAS 파싱 실패로 이어진 것으로 추정된다.

### 수치 해석 주의 — 생존 편향

표면적 수치(Recall 0.8611, Precision 0.5544)는 IEP-1001·1002 대비 우수해 보이나, 이는 100개 중 NaN이 발생하지 않은 15개(Recall 기준) 샘플만의 평균이다. NaN 샘플 대부분은 평가 자체가 실패한 케이스로, 이들이 낮은 점수로 측정되었다면 전체 평균은 크게 낮아질 것이다. IEP-1003의 수치는 **"측정 가능했던 케이스에서의 성능"**으로 한정하여 해석해야 한다.

**생존 편향 보정** (NaN → 0점, 전체 100개 기준)

| 지표 | 낙관적 (NaN 제외) | 유효 샘플 | 보수적 (전체 100개) |
| :--- | :---: | :---: | :---: |
| Context Recall | 0.8611 | 15개 | **0.1292** |
| Context Precision | 0.5544 | 49개 | **0.2716** |
| Faithfulness | 0.3869 | 14개 | **0.0541** |
| Answer Relevancy | 0.5693 | 88개 | **0.5010** |

> - **낙관적 수치**: 청킹 방식 자체의 잠재적 성능. 평가 인프라(judge LLM, 컨텍스트 크기)가 개선되면 이 수치에 근접할 수 있다.
> - **보수적 수치**: 현재 파이프라인 전체의 실제 안정성. 배포 가능 여부 판단 기준으로 사용한다.
> - IEP-1001 CASE 3은 NaN 2.5개 기준으로 97개 이상 유효 샘플에서 Recall 0.8520을 기록했으므로 보수적 추정치도 **0.8307** 수준으로 유지된다. 이것이 배포 채택의 핵심 근거다.

### NaN 원인 분류 및 향후 검증 계획

NaN 발생 원인은 두 가지 가설이 경합하며, 이를 구분하는 것이 Phase 3 실험 설계의 출발점이다.

| 가설 | 근거 | 검증 방법 |
| :--- | :--- | :--- |
| **가설 A** — 컨텍스트 과부하 | 평균 청크 2,096자 × top-3 = 최대 6,000자+로 `llama3.1:8b` 처리 한계 초과 | judge LLM을 컨텍스트 윈도우가 큰 모델로 교체 후 NaN 변화 관찰 |
| **가설 B** — pdfplumber 노이즈 | 빈 줄·도표 잔재가 SemanticChunker 경계를 교란 → 불완전 청크 → RAGAS 파싱 실패 | IEP-1004에서 Docling으로 파서 교체 후 동일 조건 재실험 |

두 가설은 상호 배타적이지 않으며 복합 원인일 가능성이 높다. **가설 B가 주요 원인이라면** Docling 교체 후 NaN이 크게 감소하며 낙관적 수치(Recall 0.8611)에 통계적 신뢰도가 부여된다. 이 경우 배포용 청킹을 SemanticChunker로 전환하는 근거가 된다.

### 의미 기반 청킹의 가능성과 한계

SemanticChunker는 IEP-1002의 헤딩 탐지 실패라는 제약을 우회하는 데 성공했다. 비교(1.0000)와 범위 밖(1.0000) 유형에서 완벽한 Recall을 보인 것은 의미 단위 분리가 멀티 청크 검색에 유리하게 작용한 결과다. 그러나 pdfplumber 추출 텍스트의 노이즈가 청크 품질을 불균일하게 만들었다. 이 문제는 Phase 3 IEP-1004(Docling 파서 교체)에서 근본적으로 해소할 수 있을 것으로 기대한다.

### 배포용 청킹 — IEP-1001 CASE 3 최종 채택

수치상 기준(Recall > 0.8520)은 충족했지만 NaN 85개라는 심각한 불안정성을 감안하여 **IEP-1001 CASE 3(1000/200 단순 청킹, Recall 0.8520, NaN 2.5개)**을 배포용 청킹으로 최종 채택한다. IEP-1003의 의미 기반 청킹은 Docling 파서 교체(IEP-1004) 이후 재평가한다.
