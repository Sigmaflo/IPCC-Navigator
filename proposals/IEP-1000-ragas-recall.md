# RAGAS를 이용한 Context Recall 성능 측정

| 항목 | 내용 |
| :--- | :--- |
| **상태** | `Draft`
| **작성일** | 2026-03-27 |

## 동기
- RAG시스템의 신뢰성을 확보하기 위해 수치적 평가가 필요함
- 사용자가 시스템을 신뢰하고 사용할 수 있기 위해서, 그리고 내가 시스템의 부족한점을 인지하고 개선하기 위해 시작하였음

## 진행
### RAGAS 측정지표 조사
| 평가 대상 | 지표명 (Metric) | 한국어 명칭 | 정의 및 평가 핵심 | 평가 데이터 (Required) |
| :--- | :--- | :--- | :--- | :--- |
| **Retriever**<br>(검색) | **Context Precision** | **컨텍스트 정밀도** | 질문과 관련된 유용한 정보가 검색 결과의 **상위 순위**에 배치되었는지 평가 | 질문(Question), 검색된 컨텍스트(Contexts) |
| | **Context Recall** | **컨텍스트 재현율** | 정답(Ground Truth) 도출에 필요한 **모든 정보**가 검색 결과에 포함되었는지 평가 | 정답(Ground Truth), 검색된 컨텍스트(Contexts) |
| **Generater**<br>(생성) | **Faithfulness** | **충실성** | 답변이 검색된 컨텍스트에만 근거하여 생성되었는지 평가 (**할루시네이션 방지**) | 답변(Answer), 검색된 컨텍스트(Contexts) |
| | **Answer Relevance** | **답변 관련성** | 생성된 답변이 사용자의 질문 의도에 얼마나 **직접적으로 관련**되고 유용한지 평가 | 질문(Question), 답변(Answer) |
- RAGAS는 총 4개의 지표(Context Precision, Context Recall, Faithfulness, Answer Relevance)에 대해 측정 가능함
- 측정 지표 중 Context Recall이 가장 중요하다고 판단하여 먼저 측정하기로 결정
- 이유: RAG 시스템에서 생성 단계(Generator)가 아무리 뛰어나도, 검색 단계(Retriever)에서 1가지만 찾아온다면(낮은 Recall), 챗봇은 사용자에게 불완전하거나 편향된 정보를 제공할 수 있음

### 골든 데이터셋 데이터 유형 분류

| 번호 | 분류 (Type) | 정의 및 특징 | 질문 예시 (User Input) | 평가 포인트 (Context Recall Focus) |
| :--- | :--- | :--- | :--- | :--- |
| **01** | **사실 확인 (Fact Check)** | 보고서 내 특정 수치, 연도, 시나리오 결과 등 단일 팩트 추출 | "지구 온도가 2도 상승하면?" | 특정 수치가 명시된 **단일 청크(Single-chunk)**를 정확히 리트리벌 하는지 확인 |
| **02** | **비교 (Comparison)** | 서로 다른 시나리오, 지역, 시점 간의 차이점 및 공통점 분석 | "지구 온도 상승 1.5도 vs 2도 차이?"" | 두 가지 이상의 서로 다른 섹션에 흩어진 **멀티 청크(Multi-chunk)**를 누락 없이 가져오는지 확인 |
| **03** | **의견/예측 (Opinion/Forecast)** | 과학적 근거를 바탕으로 한 미래 전망이나 특정 지역의 위험도 설명 | "미래에 한국의 봄과 가을은 없어질까?" | 전문적인 수치를 **일반인용 맥락(Narrative)**으로 풀어서 설명할 수 있는 충분한 배경 정보를 가져오는지 확인 |
| **04** | **범위 밖 (Out of Scope)** | IPCC 보고서의 범위를 벗어난 정치, 윤리, 개인적 의견 요구 | "보고서에서 가장 아름다운 그림은?" | 관련 없는 정보를 가져오지 않고, **거절 응답(Refusal)**을 위한 근거(범위 미포함 확인)를 정확히 판단하는지 확인 |
- 시스템이 어떤 부분에서 취약한지를 확인하기 위해 데이터셋을 유형별로 분류하고 만들 필요가 있다고 판단하였음
- 사용자의 질문 유형을 4가지(단순 사실 확인, 2가지 이상 정보 비교, 미래 예측, 보고서에 없는 내용)로 분류함
- 유형별 데이터의 개수는 25개이고 총 데이터의 개수는 100개로 결정

### Conext Recall 지표 성능 측정
| 질문 유형 (Type) | 1회차 | 2회차 | 3회차 | 최대 편차 (Max Diff) | 비고 |
| :--- | :---: | :---: | :---: | :---: | :--- |
| **01. 사실 확인** | 0.5952 | 0.5952 | 0.6273 | **0.0321** | 유의미한 변동 발생 |
| **02. 비교** | 0.8897 | 0.8897 | 0.8897 | 0.0000 | 측정 일관성 매우 높음 |
| **03. 의견/예측** | 0.8875 | 0.8875 | 0.8875 | 0.0000 | 측정 일관성 매우 높음 |
| **04. 범위 밖** | 0.9095 | 0.8595 | 0.8595 | **0.0500** | 1회차 이후 하향 고착 |
| **전체 평균** | **0.8205** | **0.8080** | **0.8160** | **0.0125** | 전체 변동폭 1.5% 내외 |

| 질문 유형 (Type) | 1회차 NaN | 2회차 NaN | 3회차 NaN | 변동 내역 |
| :--- | :---: | :---: | :---: | :--- |
| **01. 사실 확인** | 4개 | 4개 | 3개 | 1개 항목 결과 복구 |
| **02. 비교** | 4개 | 4개 | 4개 | 변동 없음 |
| **03. 의견/예측** | 5개 | 5개 | 5개 | 변동 없음 |
| **04. 범위 밖** | 5개 | 5개 | 5개 | 변동 없음 |
| **합계** | **18개** | **18개** | **17개** | **총 1개 항목 변동** |

## 분석
- 종합적으로 0.8정도의 준수한 recall 신뢰성이 측정됨
- 가장 간단하다고 생각되는 **사실 확인** 지표의 신뢰성이 가장 낮게 나와서 원인 파악이 필요함
- 측정하지 못한 NaN데이터가 17-18개 정도 나왔는데 이것은 평가 모델의 컨텍스트 한계의 문제로 추정됨
